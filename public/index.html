Automated 2D hydraulic modeling

From live gage data to a flood inundation map, with nothing in between.

HECinBox wraps a full HEC-RAS 2D unsteady workflow in a single container. It pulls real-time USGS discharge and NOAA tide records, writes them into your model as boundary conditions, runs the HEC-RAS engine headless, then returns depth maps, animations, validation statistics, and threshold alerts. No desktop session, no manual data entry.

The problem

A forecast run should not take an afternoon

The modeling is the easy part. What burns the day is downloading gage records, reshaping them into boundary condition tables, opening the plan, clicking compute, waiting at the desk, then exporting maps by hand. Do that every six hours during an event and the workflow becomes the bottleneck.

The manual loop

  • × Download CSVs from three agency portals, fix the time zones by hand
  • × Paste flow and stage series into the plan, one boundary at a time
  • × A Windows desktop must stay awake for the whole computation
  • × Export maps and statistics manually for every single scenario
  • × Nobody is watching the results at 3 a.m. when the threshold is crossed

With HECinBox

  • ✓ Data is fetched by API and aligned to one local standard time base
  • ✓ Boundary conditions are written into the plan HDF automatically
  • ✓ The Linux engine runs headless, detached from the browser session
  • ✓ Depth maps, animations, and goodness of fit statistics come out with the run
  • ✓ Rule based agent emails the moment a threshold is crossed

Capabilities

Everything the workflow needs, in the box

Built for practicing hydraulic engineers and for research groups that need to run the same model hundreds of times without touching it.

Live boundary conditions

USGS instantaneous discharge and stage, NOAA water levels, National Water Model and STOFS forecasts, all fetched by API and injected as flow, stage, or normal depth boundaries.

Model scanner

Reads the project file and geometry HDF to find plans, 2D flow areas, mesh cell counts, and every named boundary condition line, so nothing has to be typed in twice.

Headless HEC-RAS engine

Bundled Linux compute binaries run the geometry preprocessor and unsteady solver with live progress. Choose the thread count, close the tab, the run keeps going.

Inundation mapping

Filled mesh cells rendered on the GPU, depth class bands the way flood hazard maps present them, shaded relief draping, plus satellite and terrain basemaps and a 3D view.

Flood propagation video

Every time step composited into a playable animation of the flood spreading across the domain, with a seek bar and a download for presentations.

Validation built in

Modeled series compared against the observed gage record with NSE, RMSE, MAE, bias, and a one to one scatter, so calibration quality is visible on the same screen as the result.

Threshold agent

Define rules on a cell depth, a domain peak, or wetted area. Each finished run is evaluated and an email goes out the moment a condition crosses from safe to exceeded.

Continuous scheduling

Set an interval and the pipeline re-fetches, re-runs, and re-publishes on its own. A read only live view can be shared with people who should watch but not touch.

Cloud native storage

Read the model from S3, Cloudflare R2, MinIO, or Wasabi and write finished runs back. Credentials come from the environment or an attached IAM role, never from the browser.

Rain on grid

Direct precipitation on the mesh from a constant rate, an AORC or HRRR gridded product, or an uploaded DSS file. The forcing is baked into the plan before the run starts.

Nine tab workspace

Model, window, boundaries, run, results, validation, agent, live, and a full user manual. Every panel is a step in the same pipeline, in the order an engineer would work.

Runs where you need it

One image for a laptop, an on premise server, or a Fargate task. Air gapped installations work the same way, minus the live data feeds.

Deploy

One command, then open a browser

The image is public on Docker Hub. Mount the folder that holds your HEC-RAS project, pick how many cores the solver may use, and go to localhost.


      


      

        

          
1 Mount your model

          
The container reads the project read only and writes finished runs to an output folder you control. Nothing is uploaded anywhere unless you configure cloud storage yourself.

        

        

          
2 Give it cores

          
The Run tab detects available CPUs and lets you choose how many threads HEC-RAS uses. More cores means a shorter unsteady computation on large meshes.

        

        

          
3 Scale it up later

          
The exact same image runs as a Fargate task or on an EC2 host behind a load balancer, reading models from a bucket and writing results back with an IAM role.
